## Pre-Staging Gate

Do not ask Pagely or wp-admin to upgrade production until these are captured:

- Fresh Pagely backup exists, exact timestamp is recorded, and restore path/SLA is known.
- Staging app URL is known and production has been cloned or refreshed into staging.
- Host confirms PHP 8.3+ if possible, or at minimum PHP 7.4+ with a plan to move to 8.3+.
- Host confirms MariaDB 10.6+ or MySQL 8.0+ if possible.
- Current production plugin inventory is captured from authenticated REST; confirm in wp-admin/Pagely only if support sees drift.
- Pagely explains whether blocked background updates/non-writable core files are expected for their managed update flow.
- Pagely explains whether page-cache Site Health warnings are expected under PressCache or need remediation before 7.0.
- Production-write pause window is selected. Earliest acceptable window from the plan is Sunday, 2026-05-24 PT, only if staging is green.

## Staging Procedure

1. Create or refresh staging from production via Pagely Atomic/Pagely Sync.
2. Confirm staging `siteurl`/`home` and search-replace are correct for the staging host.
3. Disable production-only behavior on staging:
   - outbound marketing emails,
   - payments/renewals,
   - public indexing,
   - any production webhook destinations.
4. Run baseline:

```bash
BASE_URL=https://staging.example.com EXPECT_VERSION=6.9.4 make wp7-smoke
```

5. Upgrade staging to WordPress 7.0 through Pagely/wp-admin.
6. Run `/wp-admin/upgrade.php` if prompted.
7. Clear staging cache.
8. Run:

```bash
BASE_URL=https://staging.example.com EXPECT_VERSION=7.0 make wp7-smoke
```

9. Complete manual wp-admin checks:
   - login, Dashboard, Updates, Site Health;
   - block editor opens and saves a private/create-only test draft;
   - revisions screen opens on a safe draft/page;
   - media library opens and uploads are visible;
   - Contact page form renders but is not submitted unless KK approves;
   - Jetpack, Site Kit, Popup Maker, Redirection, Code Snippets, and Zero BS CRM admin screens open;
   - `Settings > Connectors` is inspected only; no AI provider keys are added.

## Production Go/No-Go

Go only if all are true:

- Staging public smoke passes with `EXPECT_VERSION=7.0`.
- Manual wp-admin checks pass.
- No PHP fatal errors or critical Site Health regressions are visible.
- Contact/search/sitemap/REST/schema/OG surfaces are stable.
- Fresh backup and Pagely rollback path are confirmed.
- KK approves the production window.

No-go if any are true:

- Catch Responsive, Jetpack, Popup Maker, Site Kit, Redirection, Code Snippets, or Zero BS CRM breaks on staging.
- Staging shows public fatal errors, broken navigation, broken forms, broken search, or missing REST.
- Pagely recommends waiting for 7.0.1 due to host-side findings.
- Rollback path is ambiguous.

## Production Rollout

1. Pause content publishing and production admin edits.
2. Confirm latest backup in Pagely Atomic and record backup timestamp.
3. Run:

```bash
make wp7-smoke EXPECT_VERSION=6.9.4
```

4. Upgrade production through Pagely/wp-admin.
5. Run `/wp-admin/upgrade.php` if prompted.
6. Clear Pagely PressCache/CDN.
7. Run:

```bash
make wp7-smoke EXPECT_VERSION=7.0
```

8. Run the same manual admin checks used on staging.
9. Monitor for 24 hours, then again after 7 days.

## Rollback

- If public pages show fatal errors, stop changes, capture URL/error/screenshot, and ask Pagely to roll back to the recorded pre-upgrade backup.
- If a specific plugin breaks wp-admin or public output, deactivate only that plugin on staging first; use production deactivation only if approved and rollback is documented.
- If cache shows mixed old/new assets, clear Pagely PressCache/CDN before escalating.
- If production cannot be stabilized quickly, request Pagely snapshot restore and consider a temporary WordPress version hold.
